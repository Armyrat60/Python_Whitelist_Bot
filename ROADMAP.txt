  Last updated: 2026-03-28

VISION
------
A full platform for Squad clan and server owners to manage whitelists, verify
player identities, track stats, and run server operations — all Discord-native.

Three products, one shared data layer:
  1. Whitelist Manager  (current — Python bot + Next.js)
  2. Stats Platform     (Phase 4 — TypeScript, separate site/service)
  3. SquadOps           (Phase 5 — github.com/Armyrat60/SquadOps)

Tech stack direction:
  - Python bot stays for Discord event handling (shrinks over time)
  - New backend features built in TypeScript/Node (Fastify + BullMQ + Prisma)
  - Next.js frontend stays, expands
  - Python bot becomes a thin Discord interface, calls TypeScript API for new features

  [ ] "Verify with Steam" flow on panel submission page

2D — SquadJS Sync
  [ ] Per-org SquadJS DB connection string configured in Settings (encrypted)
  [ ] BullMQ sync job — incremental, pulls Player + SteamUser + PlayerConnected
  [ ] Populates player_identities and player_names
  [ ] Sync status visible in admin dashboard (last synced, record count, errors)
  [ ] Manual sync trigger from dashboard
  [ ] Graceful handling for orgs with no SquadJS DB (standalone clan path)
  [ ] Nightly job — backfill EOS IDs for whitelisted players that still have none

  NOTE: Each org configures their own SquadJS MySQL DB (read-only credentials).
  Orgs with no server (e.g. standalone clans) skip SquadJS sync entirely.
  BattleMetrics API supplements for those orgs.

  SquadJS DBLog tables we import from:
    Player            — steamID, eosID, lastName  (the gold table)
    SteamUser         — Steam profile data, avatar, country
    PlayerConnected   — session start/end timestamps
    PlayerDeath       — kills, deaths, weapons (used later for stats)
    Wound             — wounds dealt/received
    Revive            — revives given/received
    Match             — map, layer, duration, server

================================================================================
  ARCHITECTURE NOTES
================================================================================

Player Identity Triangle (core of everything)
  Discord ID  <-->  Steam64 ID  <-->  EOS ID
     |                  |                |
  (owned)       (Steam OpenID)     (SquadJS logs)

Data Flow
  SquadJS MySQL (read-only, per-org)
       |
  Incremental sync job (BullMQ)
       |
  Platform DB (player_identities, player_sessions, player_stats_*)
       |
  Whitelist Manager + Stats Platform + SquadOps

Per-org SquadJS DB
  - Each guild configures their own connection string in Settings
  - Stored encrypted at rest, never exposed via API
  - Orgs should create a READ-ONLY MySQL user for this integration
  - Orgs with no server skip this entirely (QG example — standalone clan)

Key DB Tables (platform layer)
  player_identities     eosID (PK), steamID, discord_id, verified flags
  player_names          steamID, name, first_seen_at, last_seen_at
  player_sessions       guild_id, steamID, eosID, connected_at, disconnected_at
  player_discord_links  discord_id, steamID, verified_at, method
  guild_integrations    guild_id, squadjs_db_url, bm_api_key, last_sync_at
  player_stats_daily    guild_id, steamID, date, kills, deaths, revives, playtime
  player_weapon_stats   guild_id, steamID, weapon, kills, deaths
  player_map_stats      guild_id, steamID, map, layer, matches, wins, kills
  special_kill_rules    guild_id, rule_name, weapon_contains, victim_steamID,
                        reward_discord_role_id, announcement_channel_id
